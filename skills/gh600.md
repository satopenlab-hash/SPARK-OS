# GH-600 Agentic Systems Skill

## Purpose

Use this skill for agent architecture, tools, memory, planning, evaluation, orchestration, guardrails, and human approval gates.

## Default operating model

Start with a **closed single-agent loop**.

```text
Goal → Plan → Build → Check → Human Review → Improve → Document → Stop
```

Do not start with a fleet or open-ended swarm.

## Escalation ladder

### Level 1 — Closed single loop

Use for:
- one document;
- one research brief;
- one mission;
- one rubric;
- one synthetic example.

Requirements:
- clear objective;
- bounded context;
- quality checklist;
- stop condition;
- human reviewer.

### Level 2 — Controlled multi-agent loop

Use only when the task genuinely has separable specialties.

Minimum roles:
- **Orchestrator:** owns task decomposition and status.
- **Maker:** produces the artifact.
- **Checker:** evaluates independently.
- **Human approver:** decides whether to accept, revise, or stop.

### Level 3 — Open fleet

Not permitted in Genesis or early Pilot stages without:
- explicit budget;
- tool allowlist;
- logging;
- safety gates;
- recovery plan;
- named human supervisor;
- clear stop conditions.

## Agent outputs must state

```text
Goal:
Inputs used:
Actions taken:
Tools used:
Output:
Confidence:
Assumptions:
Missing evidence:
Failure risks:
Human approval required:
Stop condition:
Audit trail location:
```

## Tools and MCP-style access

Treat every tool as an authority boundary.

Before use, define:
- allowed data;
- prohibited data;
- action scope;
- expected output;
- logging requirement;
- who can approve side effects.

## Memory rules

- Prefer repository memory over hidden or transient memory.
- Store durable decisions in Markdown or JSON.
- Never store sensitive student information in public repositories.
- Distinguish facts, interpretations, assumptions, and unresolved questions.

## Evaluation principle

The maker should not be the only checker.

For material outputs, use an independent review pass or human review.

## Common GH-600 failure modes

- agent goal is vague;
- tool permissions are too broad;
- no stopping rule;
- checker repeats maker errors;
- hidden context prevents reproducibility;
- a prototype is mistaken for a reliable system;
- the human is invited only after irreversible actions occur.
